# 出院患者联系方式补录与回访展示设计

## 背景

当前系统已经支持出院患者导入和患者回访，但 `DischargedPatient` 模型中没有电话、住址字段，导入链路也只覆盖姓名、病案号、诊断、出入院日期、主管医师等基础信息。

用户希望将现有文件 `D:\我开发的项目\科室业务平台V3\2024-2025病案信息.xlsx` 全量导入到现有患者库中，用于补充患者电话和住址，并在“患者回访”页面直接显示这些信息，方便回访操作。

用户已确认：

- 导入模式为“保留现有数据，更新/补充联系方式”，不是清库重建
- 匹配规则优先 `病案号`，找不到时回退到 `患者姓名 + 出院日期`

## 目标

- 支持从病案 Excel 中识别并导入患者电话、住址
- 在不清空现有患者数据的前提下，按已确认的规则更新或补充联系方式
- 在患者回访页面直接展示电话和住址
- 兼容当前出院患者导入入口，不新增独立导入流程

## 非目标

- 不修改回访业务流程、权限规则、统计口径
- 不新增独立“联系方式补录导入”页面
- 不要求用户必须先改 Excel 模板
- 不处理超出电话、住址之外的其他病案字段补录需求

## 现状

### 数据层

`server/app/models/patient.py` 中 `DischargedPatient` 当前字段包括：

- `patient_name`
- `record_no`
- `gender`
- `age`
- `diagnosis`
- `admission_date`
- `discharge_date`
- `attending_doctor`
- `import_batch`
- `imported_at`

缺少：

- `phone`
- `address`

### 导入层

`server/app/utils/excel_import.py` 中：

- `STANDARD_FIELDS` 未包含电话、住址
- 自动匹配仅依赖标准中文列名
- 行转换逻辑支持字符串、日期、整数，但没有联系方式字段语义

### 展示层

`frontend/src/views/PatientFollowup.vue` 当前待回访列表显示：

- 患者姓名
- 病案号
- 诊断
- 出院日期
- 主管医师
- 医师/护士回访状态

回访弹窗顶部仅显示：

- 患者姓名
- 诊断
- 出院日期

没有电话和住址展示位。

## 方案概述

在现有“出院患者导入”链路上扩展电话与住址字段，导入时对已有患者执行增量更新，并在“患者回访”列表与回访弹窗中展示联系方式。

### 方案摘要

1. 在 `DischargedPatient` 中新增 `phone`、`address` 字段
2. 扩展 Excel 字段映射，兼容常见表头别名
3. 修改患者导入逻辑为“优先更新匹配患者，否则新增”
4. 在患者回访页面增加电话、住址显示

## 详细设计

### 1. 数据模型变更

在 `DischargedPatient` 中新增：

- `phone = db.Column(db.String(64), default="")`
- `address = db.Column(db.String(255), default="")`

并在 `to_dict()` 返回结构中加入：

- `phone`
- `address`

#### 兼容性

- 对 SQLite 现有库执行轻量 schema 补充
- 对已有数据默认填空字符串，不影响旧页面读取

### 2. Excel 字段识别

在 `server/app/utils/excel_import.py` 中扩展标准字段与别名匹配。

新增标准字段：

- `phone`
- `address`

建议支持的表头别名：

#### 电话

- `电话`
- `联系电话`
- `联系方式`
- `手机号`
- `手机号码`

#### 住址

- `住址`
- `家庭住址`
- `现住址`
- `联系地址`
- `地址`

#### 规则

- Excel 中有值时导入该字段
- Excel 中缺少电话或住址列时，本次导入仍然成功
- Excel 中某行该字段为空时，不强制覆盖已有值

### 3. 导入更新策略

导入行为从“直接新增”调整为“匹配更新或新增”。

#### 匹配顺序

1. 优先按 `record_no` 匹配已有患者
2. 若 `record_no` 为空或未匹配，回退到 `patient_name + discharge_date`
3. 两者都未命中时，按现有逻辑新增患者

#### 字段写入策略

- Excel 中 `phone` 有值时，更新到患者记录
- Excel 中 `address` 有值时，更新到患者记录
- Excel 中 `phone/address` 为空时，不覆盖数据库中已有非空值
- 其他基础字段继续保持现有导入逻辑，除非当前实现已限定只在新增时写入

#### 导入结果建议

接口返回建议包含：

- 导入总行数
- 新增数
- 更新数
- 跳过数

即使本次不调整前端统计文案，后端内部也建议保留新增/更新计数，便于排障。

### 4. 患者回访页面展示

修改 `frontend/src/views/PatientFollowup.vue`。

#### 待回访列表

新增两列：

- `电话`
- `住址`

建议布局：

- 电话列宽固定，便于快速拨打
- 住址列使用 `show-overflow-tooltip`，防止表格过宽

#### 回访弹窗

在患者信息摘要区增加：

- 电话
- 住址

展示目标是让操作人打开弹窗即可看到联系方式，无需返回列表确认。

### 5. 数据库初始化/升级

当前项目已经在 `server/app/__init__.py` 中处理 SQLite 索引初始化，说明项目接受应用启动时做轻量 schema 修复。

本次建议沿用同样模式：

- 启动时检测 `discharged_patients` 是否存在 `phone`
- 启动时检测 `discharged_patients` 是否存在 `address`
- 若不存在，执行 `ALTER TABLE`

这样不引入额外迁移工具，符合当前项目结构。

## 错误处理

- Excel 识别不到任何核心患者字段时，仍按现有行为报错
- Excel 缺少电话、住址列时，不报错，只是不更新这两个字段
- `record_no` 与姓名/出院日期都无法匹配时，走新增
- 同名同出院日期存在多条命中时，应优先要求病案号命中；回退匹配若存在歧义，建议只取第一条并记录日志，避免整个导入失败

## 测试方案

### 后端

- 导入包含电话、住址列的 Excel，验证能正确解析
- 已有患者按病案号匹配，验证电话、住址被更新
- `record_no` 为空时，按姓名 + 出院日期匹配成功
- Excel 电话/住址为空，验证不覆盖数据库中的非空值
- 缺少电话/住址列时，导入仍成功

### 前端

- 患者回访待办列表显示电话、住址
- 地址超长时 tooltip 正常
- 回访弹窗可看到电话、住址
- 无电话/住址时显示为空，不影响回访提交

### 联调

使用 `2024-2025病案信息.xlsx` 实际执行一次导入，确认：

- 已有患者联系方式被补录
- 回访页列表可直接查看电话和住址
- 导入不会因为个别空值中断

## 风险与权衡

### 风险 1：Excel 表头不统一

通过别名匹配降低风险，但仍可能遇到特殊表头命名。设计上选择“尽量识别，识别不到则忽略联系方式字段”，避免整批失败。

### 风险 2：回退匹配存在歧义

`患者姓名 + 出院日期` 不是强唯一键。设计中将其作为病案号缺失时的兜底匹配，而不是主规则，以降低误更新概率。

### 风险 3：旧数据覆盖

为避免把已有联系方式洗掉，约定“Excel 空值不覆盖库内非空值”。

## 推荐实施顺序

1. 扩展模型字段与 `to_dict`
2. 补 SQLite 启动时 schema 修复
3. 扩展 Excel 字段映射与别名
4. 修改导入更新逻辑
5. 修改回访页列表与弹窗展示
6. 用真实 Excel 联调验证
